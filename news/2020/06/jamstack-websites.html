What is JAMStack and how does it differ from static website generators?

  1. AkashaCMS - the powerful static file CMS to build your dream website
  2. Announcements and News about AkashaCMS
  3. AkashaCMS News for 2020
  4. News for June 2020
  5. What is JAMStack and how does it differ from static website generators?
Date: June 27, 2020
By David Herron

Two ideas, JAMStack and static website generators, are getting attention as a way to simplify website and web application development, while simultaneously being vague unclear marketing terms. Both talk about static assets for website, but what does that really mean? To help decode the vague marketing words, let's examine the technologies, see how they differ, and try to figure out if there's anything important to learn.

In the beginning (of the web) websites were HTML files (and images) you uploaded to a web server using FTP. That's it. My first websites, in 1994-1995, were coded that way, and of course I used FTP. A big advance was finding a programmers editor, that handled HTML syntax coloring, and which could directly edit files in remote directory structures. That meant not having to upload using FTP. Another advance was a feature called Server Side Includes, letting one have sidebar or header or footer "widgets" loaded as HTML snippets from a shared directory. That simplified having a consistent look across the website. Then came the variety of pseudo-WYSIWYG HTML editors where we could edit web pages visually, rather than guessing what the page looked like while editing raw HTML. These editors didn't support server side includes, but there was a similar feature in Dreamweaver allowing one to have shared HTML snippets.

The point is that a website built with simple HTML files is not a new concept. We've been doing that since the beginning of the Web.

The story for both JAMStack and Static Website Generators is that there's a big speed advantage with static HTML+CSS+JavaScript rather than deploying a dynamic content management system (CMS) or a custom server-side application. There's no complex server-side code rendering pages.

For me, both are a reaction to complicated dynamic CMS's, or complex custom micro-services deployed to cloud hosting. Over the years of learning the intricacies of Drupal 6 and Wordpress, learning how to customize Blogger templates, and learning Docker and Docker Swarm, I came away with a distaste of the burden it is to host a complex content management system which dynamically generates pages.

At one time I was maintaining 10 or so Drupal instances for different websites. That led to multiple VPS's, a never ending grind of updating modules on each site, making sure nothing had broken, and worrying how I would pay the hosting bill.

Instead I prefer deploying static HTML+CSS+JS to a simple server. Instead of doing that the old way, I see no reason why deploying static files to a simple server means we can't use modern HTML+CSS+JS.

As a result of those experiences, I developed my own system, AkashaCMS. It is a static website generator, giving me a certain perspective on this whole question...

What is JAMStack?

The JAMStack concept is defined as "Fast and secure sites and apps delivered by pre-rendering files and serving them directly from a CDN, removing the requirement to manage or run web servers." To drive the point home the acronym, JAM, means:

  • (J) JavaScript executed in a web browser page as the client tier
  • (A) API's hosted on servers are accessed by client-side JavaScript to retrieve data
  • (M) Markup, both as HTML and HTML templates, are used to dynamically update pages using the JavaScript and retrieved data
  • (STACK) Because, well, every software product is described as a Stack, right?

That's simple enough, static assets, but the definition says "sites or apps" and requires deployment to a CDN. What's the real difference between this and a static website generator?

In a system like AkashaCMS, the user edits Markdown or AsciiDoc files, renders them to HTML, and uses rsync to deploy to a web server. Using modern HTML+CSS+JS simply requires suitable layout templates and the use of a modern CSS+JS framework to handle modern responsive layouts. Other than the CDN, is there a difference? In both cases pre-rendered HTML+CSS+JavaScript are deployed to a hosting location, and the hosting location does nothing other than deliver the files to web browsers.

With both, we skip a bunch of complexity and cost around dynamically generating assets via dynamic back-end services.

Except ... The JAMStack proponents insist on that "serving them directly from a CDN" phrase. A CDN, a.k.a. Content Delivery Network, is a cloud based service for distributing assets to remote servers around the world. That way your visitors will always receive files from a nearby CDN endpoint, rather than having to retrieve files from your webserver.

But, is there any practical difference to using a CDN?

Why insist on a CDN?

In the JAMStack literature the task of "managing a webserver" is described as a huge scary burden. The CDN is described as being "serverless", which is rather silly because of course CDN's are built from zillions of servers. What "serverless" really means is that the person deploying the code isn't involved with the overhead of provisioning and managing servers. With a CDN, you upload the files, and the CDN takes care of both the servers and distributing your HTML+CSS+JS assets around the world.

Whats meant by the word Serverless is that we (the customer) don't have to configure the servers. We just deploy our code, and the serverless service takes care of the rest.

The advantage for a CDN probably comes when you have a LOT of traffic.

According to Google Analytics the page load speed for my websites is under 20 milliseconds, even though the static HTML+CSS+JS is hosted on a simple Apache-based shared hosting account. Seems to me that 20 milliseconds is pretty close to blazingly fast. I imagine that if one of my sites were receiving 100,000 visitors a day, Dreamhost would be sending me emails about the load on their servers. CDN's automatically distribute the assets to a network of servers around the world, and have performance guarantees that cannot be made by the traditional Apache-based web hosting provider.

Where did JAMStack come from?

It seems that JAMStack came from Front End Engineers who had been roped into the unnatural thing of standing up Node.js services as the back end of their web application. Someone whose skillset is writing HTML, CSS and JavaScript, might be expected to find Node.js development familiar because it's JavaScript.

But, really, the coding experience of front end HTML+JavaScript+CSS is very different from the back end experience on a server. Even of the back end code is JavaScript, like it is with Node.js, the two are very different tasks. For example front-end engineering is oriented to visual and interactive effects, whereas back-end coding is about server deployments, service scaling, and the like.

It stands to reason a front-end engineer would balk at doing server deployments.

The folks talking about JAMStack discuss how easier it is to deploy applications. The folks using static website generators, like me, are looking for a slightly different thing. We have websites (blogs etc) to deploy.

Avoiding complexity may be the biggest gain

This image demonstrates the advantage of both JAMStack and the websites rendered using a Static Website Generator. The far simpler system is lower cost and lower resource impact and lower overhead to maintain and faster.

On the left (blue arrows and text) we have the model of deploying static assets. They can be deployed to a CDN, as the JAMStack proponents insist, or to a simple Apache or NGINX server. In both cases the user's web browser makes a request to a web server, and the web server simply sends back the file, and that's that.

On the right (with green text and arrows) we have a complex system deployed on cloud hosting infrastructure. A cloud hosting platform like AWS has close to a zillion features, giving technologists lots of things to get excited about. Global scale custom services of the type driving all kinds of websites are possible by configuring and using cloud hosting platforms. It's a compelling model, because anyone can rent service that can scale from a small deployment to a globe spanning service at the click of a few buttons. But trust me when I say that it is complex to learn, complex to understand, complex to grok how to minimize the costs, complex to manage, etc.

That's nice theory, but what is JAMStack or static websites for real?

All of that was rather abstract, so let's look at some concrete examples of JAMStack.

At the simplest, JAMStack is an HTML page, containing JavaScript, where it makes API calls to services, using retrieved data to make changes to the HTML.

Commenting systems on static HTML or JAMStack websites

There were advantages to those dynamic CMS's. For instance Wordpress and Drupal both integrate a native commenting system, and allow users to register with the website in order to leave comments. It is possible to build a community via a website by allowing your readers to leave comments. Why should we leave that and other features behind?

Let's concentrate on what it would take to implement a commenting system.

The first step is adding some JavaScript to each page where you want comments. The JavaScript would make calls to a back-end commenting system, and also render a user interface in a given location in the web page. For example:

var dsq = document.createElement('script'); dsq.type = 'text/javascript'; dsq.async = true;
dsq.src = 'https://' + disqus_shortname + '.disqus.com/embed.js';
(document.getElementsByTagName('head')[0] || document.getElementsByTagName('body')[0]).appendChild(dsq);

This is the JavaScript provided by Disqus, a large scale commenting service. It handles comments on websites, and has many useful tools for managing spammers and user authentication.

The Disqus UI integrates a number of back-end services. For example users can sign up as a Disqus user without leaving the web page, and it can also authenticate people using 3rd party services like Facebook or Twitter. Once a reader is authenticated, they can leave comments and the comments area is dynamically updated as others leave comments.

That's one example, a 3rd party service providing some kind of service which adds information to the web page. You could imagine other services like weather forecasts, or auction website searches, and more. You can see most advertising networks as an example as well, since they also provide an HTML+JS snippet that calls a backend service and then manipulating the DOM to include the advertisement.

Custom back-end services in static HTML or JAMStack websites

Many websites have custom services as well. Not every need fits into what a 3rd party service can provide, after all.

To take an example, one of my websites has a service implemented in old school PHP. The original version of that service was implemented in TCL in 1997, and sometime in the 00's I rewrote it to simplistic old-CGI-style PHP. Today, the PHP scripts are rendered from templates by AkashaCMS and deployed with the HTML+CSS+JS of that website.

That website is therefore a hybrid where some parts of the site are rendered dynamically. On visiting the service, the PHP script runs to generate the content, and the users can register with the website to leave information. Over 10,000 readers of the site have registered over the years.

Because it is a hybrid with dynamic rendering, that website does not strictly speaking fit the JAMStack model. Let's think about how to rectify that issue.

For example, an AWS Lambda function would be a nice way to implement the service, and then deploy the user interface as a Vue.js application. It would require an authentication system between the front end application and the deployed back-end service - I would not want a 3rd party site to hijack this service. Hosting for the back-end service is simplified by using a Functions-as-a-Service platform like AWS Lambda.

The front end application(s) can be easily written in Vue.js (or ReactJS etc) and rendered to static JavaScript for deployment along with the rest of the website.

A couple years ago I developed an example on one of my other sites of using Webpack to build Vue.js application code that can be deployed with a static HTML website. It's relatively easy, just add the Webpack execution to the build procedure for your assets, making sure the output lands in the directory where all the built assets live.

Outlining a JAMStack app website

Instead consider a website showing pollution measurements at different monitoring stations around a major city. The site could have a map with markers for the location of each monitoring station, and clicking on a marker would show a popup window with pollution statistics. There are a couple such sites in existence, and in our household we turn to one every fall because the large number of wildfires in California every year that makes the air literally unhealthy.

For such a site to be JAMStack, we'd need these features:

  • Static HTML for the overall site
  • Static JavaScript code bundle containing the logic required to render that map and all the other features required for the service
  • The JavaScript for that service obviously will query both custom back-end services, as well as third party services
  • The static HTML and JavaScript must be hosted via a simple webserver or a CDN

While the JAMStack ideology is serverless, there are back-end services for any API's utilized in the specific application. Those services obviously must be hosted somewhere. The team fielding that air quality monitoring site I just outlined would be identifying any 3rd party API's they'd use, plus developing any custom in-house-developed API's.

Typical static HTML websites might not literally be JAMStack

Let's talk about another kind of site, those built with static website generator packages. In the case of an AkashaCMS site, the author creates a Markdown or AsciiDoc file for each page on the site. That's what I'm doing to write this post - I use Visual Studio Code to edit the Markdown, and have a Markdown previewer showing a preview as I work. When I think the content is ready, I run an AkashaCMS command to render the Markdown files to HTML, and then I use rsync to upload the site to a server that runs Apache.

For most AkashaCMS-based sites there is no backend service, since it's just HTML using the Bootstrap framework (that involves some JavaScript) for theming and layout. The only JavaScript+API tool is Disqus for commenting.

Therefore the AkashaCMS sites I've built so far aren't technically JAMStack, because there is very little manipulation of the HTML by JavaScript that calls an API. Yes, these sites are using Disqus, which has JavaScript calling an API to do DOM manipulation, but we just worked out that Disqus is probably not JAMStack compliant.

One of the sites I could redesign with the JAMStack ideology in mind. It has a large directory of people performing a certain service. Currently that directory is implemented using simple plain PHP created over 20 years ago. But, converting that PHP to an AWS Lambda service that sports an API, and calling that API using client-side JavaScript, would make it a JAMStack directory application.

In recent years the website development field has been abuzz with two ideas. First it was static website generators, and more recently it was JAMStack. The curious thing is that since the two seem to be the same idea, why is one targeted at website development and supplanting Wordpress or Drupal, while the other targets a simpler process of hosting the front end of web application?

In both the idea is to pre-render the content, deploying static HTML, CSS and JavaScript to a hosting server. The idea is to skip a bunch of complexity and cost around dynamically generating assets, and instead use low-cost web hosting or to host assets on a CDN or a service like AWS S3.

The JAMStack concept is defined as "Fast and secure sites and apps delivered by pre-rendering files and serving them directly from a CDN, removing the requirement to manage or run web servers." To drive the point home the acronym, JAM, means:

  • (J) JavaScript executed in a web browser page as the client tier
  • (A) API's are accessed by the JavaScript to retrieve data
  • (M) Markup, both as HTML and HTML templates, are used to dynamically update the page driven by the JavaScript and retrieved data
  • (STACK) Because, well, every software product is described as a Stack, right?

Some JAMStack proponents insist that the static assets for a JAMStack site must be deployed solely to a CDN. But, really, is that necessary? Can't a regular web host running Apache do a good job with such assets?

To me, that begs the question of just how does a JAMStack site differ from a static website generator site? With a static website generator, we run a tool to generate HTML, CSS and JavaScript files, then upload those files to a simple web server. That's exactly what the JAMStack proponents are doing.

Before you object that a static website generator is just deploying HTML with no room for dynamic JavaScript on the client, stop and think for a moment. A trivial example is the question of providing commenting on a statically generated website. With a dynamic CMS like Wordpress or Drupal, there is a built-in commenting system. With a static website, there's no back end, which is a huge advantage, so how do we handle comments? Simply by installing a JavaScript client, that invokes API's on a commenting service, and which dynamically updates comments on the page. The primary example of this is Disqus.

It's quite feasible to add any other kind of front end client on the HTML of a statically generated website.

My story - Drupal and Wordpress admin - developer of static website generator called AkashaCMS

I've been publishing websites since 1994, with my first significant site starting in 1995. Of course at that time it was hand-crafted HTML, and was rather clunky. I still maintain a version of that site, and over the years have published a number of other sites.

In the mid-00's I grew rather fond of Drupal. I learned Drupal 6 inside and out, and was fascinated with the power of the Views module. In case you don't know, Views is a way to build queries over the content in a website, generating lists or tables or whatnot of the results. It's extremely powerful, so much so that in Drupal 8 it became part of the core of Drupal. At the time it seemed powerful and flexible to customize every page even to the point of giving each user their own experience of the site. Drupal can show registered users different themes or sidebar widgets based on their user privileges, or the users own settings.

But, Drupal, and Wordpress, and other dynamic CMS's, come with the cost of a complicated server. In most cases high performance requires caching, which adds to complexity.

By 2012 or so, I grew tired of the cost of the VPS hosting my Drupal (and Wordpress) websites. That led me to develop AkashaCMS, which is a static website generator that I think is pretty powerful.

My inspiration was exactly what I hear the JAMStack and Static Website Generator proponents argue. The kind of sites I publish do not need much customizability, so there's no advantage to hosting a dynamic CMS. The dynamic CMS requires extra complexity in the server, and the long page load times mean suffering users. The search engines are nowadays using page load time as a ranking signal, and the simplest way to decrease that time is to statically render the HTML.

JAMStack & Static Website Generators are two different things seemingly, despite being almost exactly the same

JAMStack proponents are targeting a simpler experience of deploying applications - by which they mean a JavaScript app running in HTML. Static Website Generator proponents instead target a simpler experience of hosting regular websites, and supplanting tools like Wordpress or Drupal.

JAMstack is comprised of:

JavaScript execution on the client-side (J) APIs to interact with the backend (A) Markup templates prebuilt at deploy time, using static site generators or build tools (M) A JAMstack-ready CMS, as according to Carson Gibbons, co-founder of Dallas-based Cosmic JS, is “a CMS that makes content available via API, ready to plug into the JavaScript-powered frontend.”

JAMStack is starting to be "a thing", it seems. A few months my news feed started having articles pushing JAMStack this and JAMStack that as the new Savior, a new technology that would free us from complicated something or other. The descriptions were vague, but seemed in line with the same ideas that led me to develop AkashaCMS (A static website generator application). Namely, the value of prerendering HTML and other assets as static files in order to improve page load times and the user experience.

Having now read up on the specifics of JAMStack, I'm wondering what the big deal is. There's not much in the way of technological prowess required to implement JAMStack. Yes there's some good ideas, but they're hardly earth-shattering. For example I came up with very similar ideas on my own in 2012, and the existence of the dozens of other static website generator tools indicate that lots of others have had the same inspiration. As the author of a static website generator tool, I have a couple things to discuss about JAMStack.

Let's start with the definitions so we know what we're talking about. On jamstack.org the idea is described as:

Fast and secure sites and apps delivered by pre-rendering files and serving them directly from a CDN, removing the requirement to manage or run web servers.

That's nice, prerendered files, decreased complexity, and serving files from a CDN. How does that differ from the people recommending AWS S3 as a web-hosting platform? But we can learn something more from delving into the meaning of the name.

  • JAM means JavaScript and Markup
  • This means to have HTML pages, with modern CSS, and JavaScript making API calls, that then modify the HTML markup
  • "Stack" is simply generically saying there is a "technology stack" for developing JAMStack sites

That's nice, because obviously having prerendered HTML, CSS and JavaScript assets reduces a lot of server overhead. With modern JavaScript there is little need for dynamically rendering each page on each page view. But how does this differ from e.g. using Disqus for a commenting system on a website rendered by a static website generator?

The folks promoting the JAMStack idea are coming from a different place than I am. Before developing AkashaCMS, I'd spent several years developing websites with Drupal or Wordpress. The JAMStack proponents instead were front-end engineers developing applications, and frustrated over the difficulties of standing up the servers required to deploy their applications. Where my focus is on authoring blog posts, their focus is on application deployment.

Which begs the question of how does this differ from old-school HTML-only websites? In the bad old days we'd use software like Dreamweaver, or other HTML editors, whether WYSIWYG or not, and create HTML files directly. A big advance was made with dynamic website tools like Wordpress or Drupal, in that the HTML pages were prebuilt and you could focus on the content.

With a dynamic CMS, for every page view the page is rendered from templates and data from databases. This carries a large overhead and the requirement to keep a web server running, at a large cost. Hence the part of the JAMStack definition talking about managing and running web servers.

But it seems the core idea for JAMStack is HTML files on some kind of web server. Since that's the case, what's the big deal with JAMStack? Oh, right, it's another of these marketing buzzwords that have little technical depth. But there are a couple differences here, such as modern tooling for generating static HTML, that make for significant differences over the bad old days of coding the HTML for each page from scratch.

I come to this discussion having grown disillusioned with dynamic CMS's (after doing LOTS of work with Drupal and Wordpress) because of the server overhead. As a solopreneur I want to decrease my hosting costs (avoiding the cost of running a dynamic CMS), and as a green technology advocate I want to decrease the energy/resource impact of my actions.

Due to those and other reasons, I developed AkashaCMS, which is a "static website generator". I had many of the same insights that today's JAMStack advocates discuss. The cost of managing or running a higher-end webserver is an issue for not just individuals like me, but for larger corporations. We all want to decrease costs. I realized that modern HTML doesn't have to be dynamically generated on every page view. The HTML could be prerendered, and we use modern CSS and JavaScript to add value.

For example the primary advantage for dynamic CMS's is the commenting system. But with a service like Disqus, we can get comments by using a JavaScript client. Disqus fits the JAMStack model like a glove -- because it involves installing the Disqus client code, which then makes API calls to the Disqus servers to retrieve data, that it then writes into the page.
